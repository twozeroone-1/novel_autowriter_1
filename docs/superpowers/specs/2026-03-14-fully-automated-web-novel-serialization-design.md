# 완전 자동 웹소설 연재 시스템 설계

**목표**

문피아와 노벨피아를 중심으로, `회차 기획 -> 원고 생성 -> 품질 판정 -> 플랫폼별 발행 -> 마케팅 산출물 생성 -> 정식 연재 이력 보존`까지를 무인으로 운영하는 자동 연재 시스템을 설계한다. 이 시스템의 최우선 목표는 `품질 안정성`과 `비용 절감`이며, 좋은 회차를 많이 만드는 것보다 `나쁜 회차를 절대 발행하지 않는 것`을 우선한다.

이 설계는 현행 로컬 앱 구조를 바탕으로 하지만, 단순히 프롬프트를 고치는 수준이 아니라 `운영 시스템`으로 재정의하는 것을 전제로 한다. 즉, 이 시스템은 "소설을 잘 써주는 모델"이 아니라 "저렴하게 초안을 만들고, 엄격하게 걸러내고, 안전할 때만 발행하는 컨트롤 플레인"이어야 한다.

## 배경과 문제 정의

현재 앱은 `core/context.py`, `core/generator.py`, `core/reviewer.py`, `core/automation_runtime.py`, `ui/chapters.py`를 중심으로 회차 생성과 자동화를 이미 수행한다. 그러나 지금 구조에는 완전 자동 운영에 치명적인 문제가 있다.

- AI가 쓴 회차 요약과 상태 설명이 그대로 다음 화 컨텍스트에 재주입된다.
- 요약이 `사실 기록`이 아니라 `해석과 과장`을 포함하므로, 회차가 진행될수록 설정이 증폭 오염된다.
- 검수 단계가 브레이크가 아니라 미세 수정 수준에 머물러, 생성 실패를 실질적으로 막지 못한다.
- 초안, 수정본, 리포트, 완성본의 저장 경계가 약해 파일 생명주기가 섞인다.
- 자동 연재 큐가 품질 판정과 발행 판정을 분리하지 않아, "생성 성공"과 "발행 가능"이 같은 의미처럼 취급된다.

이 문제들은 모두 같은 결론으로 이어진다. 완전 자동형에서는 `생성기`보다 `게이트`, `저장 구조`, `운영 상태`, `중단 정책`이 더 중요하다. 따라서 새 시스템은 회차 생성기를 중심에 두지 않고, `품질 게이트가 중심인 다단계 파이프라인`으로 설계해야 한다.

## 제품 정의

이 시스템은 다음 역할을 가진다.

- 플랫폼별 연재 정책에 맞춰 회차를 자동 스케줄링한다.
- 설정집과 사실 데이터베이스를 기준으로 회차를 생성한다.
- 규칙 기반 검사와 비평 기반 검사를 통해 발행 가능 여부를 판정한다.
- 기준 미달 회차는 자동 재시도하되, 정해진 한도를 넘기면 발행을 중단한다.
- 발행 성공 회차만 연재 정본으로 확정하고, 그 결과만 다음 회차 컨텍스트에 반영한다.
- 플랫폼별 소개 문구, 홍보 문구, 태그 등 마케팅 산출물을 별도 생성한다.
- 향후 해외 번역 연재를 위해 번역용 인터페이스와 데이터 경계를 미리 확보한다.

## 성공 기준

성공 기준은 "문장이 재미있다"처럼 주관적 지표가 아니라 운영 가능한 기준으로 잡는다.

- 품질 게이트를 통과하지 못한 회차는 자동 발행되지 않는다.
- 발행된 회차는 모두 `발행 전 스냅샷`, `품질 판정 결과`, `플랫폼별 업로드 이력`을 재현 가능하게 가진다.
- 회차 생성 비용은 모델 티어와 재시도 횟수로 상한을 강제할 수 있다.
- 문피아와 노벨피아는 서로 다른 스케줄과 발행 정책을 가질 수 있다.
- 하루 2연참 같은 연속 발행 정책도 정책 엔진이 제어할 수 있다.
- 향후 번역 연재는 한국어 정본을 기준으로 분리된 파이프라인으로 추가 가능하다.

## 범위와 비범위

이 설계는 범위를 의도적으로 제한한다.

`v1 범위`

- 한국어 원문 회차의 완전 자동 생성 및 발행
- 문피아/노벨피아 대상 플랫폼 어댑터
- 플랫폼별 발행 정책 엔진
- 발행 전 품질 판정과 자동 중단 정책
- 플랫폼별 마케팅 산출물 자동 생성
- 장애 감지, 런 상태 기록, 사건 대응형 운영

`v1.5 이후 확장 경계`

- 해외 플랫폼 업로드
- 번역 회차 생성
- 플랫폼별 현지화 마케팅

즉, 번역과 해외 업로드는 `이번 설계에서 반드시 고려해야 하는 인터페이스`이지만, `v1에서 바로 무인 업로드까지 완료해야 하는 구현 범위`로 잡지는 않는다. 그렇지 않으면 한 번의 구현 계획에 너무 많은 독립 서브시스템이 섞여 설계와 운영이 동시에 불안정해진다.

## 설계 원칙

### 1. 회차 본문은 진실의 원천이 아니다

생성된 회차 텍스트와 회차 요약을 다음 화의 사실로 직접 사용하지 않는다. 다음 회차의 컨텍스트는 반드시 `구조화된 사실 추출 결과`만 반영한다.

### 2. 값싼 검사부터 먼저 한다

문장 길이, 금칙 패턴, 제목/회차 번호 정합성, POV 위반, 금지된 표현, 구조 누락 같은 값싼 검사를 먼저 수행하고, 통과한 후보에만 비평 모델 비용을 쓴다.

### 3. 발행은 생성의 후속 단계가 아니라 별도의 승인 상태다

`draft_created`와 `publishable`은 다르다. 생성 성공은 발행 가능을 의미하지 않는다.

### 4. 한 회차는 하나의 거래 단위다

한 회차의 기획, 생성, 판정, 발행, 정본 반영은 동일 `run_id` 아래에서 추적된다. 어디서 실패했는지 명확하지 않으면 무인 운영은 실패한다.

### 5. 좋은 회차를 늘리기보다 나쁜 회차를 줄인다

완전 자동 시스템에서 품질 안정성은 `상한`이 아니라 `하한 관리`의 문제다. 따라서 게이트와 중단 정책이 생성 프롬프트보다 중요하다.

## 권장 아키텍처

전체 시스템은 네 층으로 나눈다.

### 1. Story Source of Truth

서사와 사실을 관리하는 기준 계층이다.

- `Story Bible`
  - 장르, 톤, 핵심 캐릭터, 금지 요소, 장기 진행 방향, 작가 의도
- `Canon DB`
  - 발행 완료 회차에서 추출된 구조화 사실만 저장
  - 인물 상태, 관계 변화, 장소, 진행 중 떡밥, 자원 변화, 시간축
- `Publishing Policy`
  - 플랫폼별 발행 시각, 하루 최대 발행 수, 2연참 허용 조건, 긴급 중단 규칙

이 계층은 사람이 편집할 수 있지만, 런타임이 자동으로 수정하는 값은 제한적이어야 한다. 특히 `Canon DB`는 오직 `발행 성공 + 구조화 추출 성공` 조건에서만 갱신한다.

### 2. Episode Factory

회차를 실제로 만드는 계층이다.

- `Scheduler`
  - 어떤 플랫폼에 언제 어떤 회차를 공급해야 하는지 결정
- `Episode Planner`
  - Story Bible, Canon DB, Release Policy를 읽고 이번 회차 목표를 구조화
- `Draft Writer`
  - 회차 본문 초안 생성
- `Quality Gate Orchestrator`
  - 규칙 기반 검사, 구조 검사, 비평 검사, 재시도 제어
- `Canon Extractor`
  - 최종 통과 회차에서 구조화 사실 추출

Episode Factory는 글을 생성하지만, 발행 여부를 단독으로 결정하지 않는다. `publishable artifact`를 만들 뿐이다.

### 3. Publish Control Plane

발행과 운영을 담당하는 계층이다.

- `Publish Packager`
  - 제목, 본문, 회차 번호, 플랫폼별 메타데이터, 예약 시각 묶음 생성
- `Platform Adapters`
  - 문피아/노벨피아 업로드 구현
- `Marketing Packager`
  - 플랫폼 소개문, 홍보 문구, 배너 문구, 요약 문안 생성
- `Incident Monitor`
  - 실패 분류, 런타임 상태 전환, 알림, 자동 중단

이 계층의 핵심은 "업로드 버튼을 누르는 것"이 아니라 "발행 결과를 신뢰 가능하게 판정하고 기록하는 것"이다.

### 4. Translation Extension Boundary

향후 해외 확장을 위한 경계 계층이다.

- 한국어 정본 회차를 기반으로 번역 패킷 생성
- 용어집, 고유명사 표기, 현지화 금칙 규칙 관리
- 번역 품질 게이트와 해외 플랫폼 업로드는 한국어 원문 파이프라인과 분리

이 경계가 없으면 번역 실패가 한국어 본편 발행 흐름을 오염시킨다.

## 핵심 컴포넌트와 인터페이스

### Story Bible Store

역할:
- 사람이 정의한 작품의 고정 규칙을 저장한다.

입력:
- 작품 설정, 장르 규칙, 캐릭터 카드, 장기 플롯, 금지 요소

출력:
- Planner가 읽을 수 있는 정규화된 설정 객체

의존성:
- 없음

### Canon DB

역할:
- 발행 완료 회차에서 확정된 사실만 저장한다.

입력:
- Canon Extractor가 만든 구조화 사실

출력:
- 회차 계획 수립용 현재 세계 상태

의존성:
- Publish Control Plane의 발행 성공 이벤트

### Release Policy Engine

역할:
- 플랫폼별 발행 규칙을 판정한다.

입력:
- 현재 시각, 플랫폼 정책, 발행 이력, 품질 상태

출력:
- `run_now`, `delay`, `stop`, `cooldown`, `burst_allowed`

의존성:
- 스케줄러, 플랫폼 이력 저장소

### Episode Planner

역할:
- 이번 회차의 서사 목적을 구조화한다.

입력:
- Story Bible, Canon DB, 연재 속도 정책, 이전 발행 이력

출력:
- 회차 목표, 등장 인물, 회수할 떡밥, 금지사항, 기대 분량

의존성:
- 없음

### Draft Writer

역할:
- 계획된 구조를 바탕으로 회차 본문을 생성한다.

입력:
- Planner 출력, 스타일 가이드, 비용 제한

출력:
- 초안 본문

의존성:
- LLM 백엔드

### Quality Gate Orchestrator

역할:
- 초안의 발행 가능 여부를 판정하고 재시도를 제어한다.

입력:
- 초안 본문, 계획 데이터, 정책 데이터

출력:
- `publishable`, `retry_possible`, `hard_fail`

의존성:
- 규칙 검사기, 구조 검사기, 비평 모델

### Canon Extractor

역할:
- 발행 가능한 회차에서 구조화 사실을 추출한다.

입력:
- 최종 통과 회차, Planner 구조

출력:
- 인물/관계/자원/떡밥/시간축 업데이트

의존성:
- 품질 통과 이벤트

### Platform Adapter

역할:
- 특정 플랫폼에 대해 로그인, 작품 선택, 회차 업로드, 발행 설정, 발행 검증을 수행한다.

입력:
- Publish Packager 산출물

출력:
- `uploaded`, `scheduled`, `published`, `failed`

의존성:
- 브라우저 자동화 세션, 자격 증명 저장소

플랫폼 공통 인터페이스는 다음 메서드를 가진다.

- `login(credentials)`
- `ensure_work(mapping)`
- `upload_episode(payload)`
- `set_publish_options(payload)`
- `verify_publication(expected)`

### Marketing Packager

역할:
- 회차 본문을 바꾸지 않고, 플랫폼별 홍보 텍스트만 생성한다.

입력:
- 발행 확정 회차, 플랫폼 규칙, 금칙어 정책

출력:
- 짧은 소개문, SNS용 문안, 태그 후보, 플랫폼용 부가 문구

의존성:
- 별도 LLM 또는 템플릿 계층

### Incident Monitor

역할:
- 실패를 분류하고 런타임 상태를 전환한다.

입력:
- 각 단계의 오류와 판정 결과

출력:
- `paused`, `blocked`, `cooldown`, `stopped`, `needs_human`

의존성:
- 모든 런 이벤트

## 데이터 저장 구조

프로젝트별 저장 구조는 `초안`, `정본`, `운영 상태`, `감사 로그`를 명확히 분리해야 한다.

예시:

- `data/projects/<id>/story_bible/`
- `data/projects/<id>/canon/`
- `data/projects/<id>/episodes/drafts/`
- `data/projects/<id>/episodes/publishable/`
- `data/projects/<id>/episodes/published/`
- `data/projects/<id>/publishing/`
- `data/projects/<id>/marketing/`
- `data/projects/<id>/runs/`
- `data/projects/<id>/incidents/`

핵심 원칙:

- 초안은 초안으로 남기고, 발행 가능한 회차와 발행 완료 회차를 다른 경로에 저장한다.
- 작품 전체 회차 순서는 `canonical episode_id`로 단일 관리하고, 플랫폼별 게시 상태는 별도 메타데이터로 분리한다.
- 회차 번호는 파일명 문자열이 아니라 메타데이터 필드로 관리한다.
- 회차 제목, 플랫폼별 제목, 발행 시각, 품질 판정은 `run snapshot`에 포함한다.
- 발행 이력은 JSONL 중심으로 append-only 로그를 유지한다.

`runs/<run_id>/`에는 다음 산출물을 남긴다.

- 입력 스냅샷
- Planner 출력
- 생성 초안
- 품질 게이트별 판정
- 최종 발행 패키지
- 플랫폼별 업로드 결과
- Canon Extractor 결과

이 구조가 있어야 사후 분석, 비용 추적, 실패 재현이 가능하다.

## 회차 파이프라인

한 회차의 표준 실행 흐름은 다음과 같다.

1. Scheduler가 플랫폼별 정책과 큐를 보고 실행 가능한 회차를 선택한다.
2. Release Policy Engine이 현재 시점에 발행 가능한지 판정한다.
3. Planner가 이번 회차의 구조화 목표를 생성한다.
4. Draft Writer가 초안을 생성한다.
5. Quality Gate Orchestrator가 값싼 검사부터 순차 적용한다.
6. 통과 후보에 한해 비평 모델 검사를 수행한다.
7. 필요하면 저비용 수정 또는 1회 재생성을 수행한다.
8. 여전히 기준 미달이면 해당 회차 발행을 중단한다.
9. Publish Packager가 플랫폼별 패키지를 만든다.
10. Platform Adapter가 업로드와 발행 설정을 수행한다.
11. 실제 발행 결과를 읽어 검증한다.
12. 성공 회차만 `published`로 승격한다.
13. Canon Extractor가 구조화 사실을 갱신한다.
14. Marketing Packager가 플랫폼별 홍보 문구를 생성한다.

핵심은 `12번 이후에만` Canon DB가 바뀐다는 점이다. 발행되지 않은 회차는 다음 회차의 진실이 아니다.

## 품질 게이트 설계

품질 게이트는 가장 중요한 설계 요소다. 완전 자동형에서 품질은 "모델이 알아서 잘 써주는가"가 아니라 "문제를 얼마나 빨리 싸게 잡아내는가"로 결정된다.

### Gate 0. 입력 정합성

- 회차 번호, 제목, 목표 분량, 등장 인물 목록, 플랫폼 정책이 모두 유효한지 검사
- Story Bible과 Canon DB가 없거나 손상되면 즉시 중단

### Gate 1. 규칙 기반 저비용 검사

- 최소/최대 분량
- 금지 표현
- 잘못된 회차 번호
- 제목 누락
- 인물명 오타 사전
- 지정 시점 위반
- 본문 구조 누락
- 과도한 반복 문장

이 단계는 가능한 한 코드로 처리한다. 가장 싸고 가장 재현성이 좋다.

### Gate 2. 구조 검사

- Planner가 요구한 핵심 장면이 실제로 들어갔는지
- 등장 인물 수가 계획과 과도하게 어긋나는지
- 이번 회차의 갈등과 변화가 존재하는지
- 다음 화 예고만 하고 실질 사건이 없는지

이 단계는 규칙과 소형 모델을 혼합할 수 있다.

### Gate 3. 비평 모델 검사

비평 모델은 하나의 점수 대신 판정과 근거를 반환해야 한다.

- `publishable`
- `retry_possible`
- `hard_fail`

검사 항목:

- 인과관계 붕괴
- 캐릭터 붕괴
- 밈 반복 과다
- 개그 인플레
- 앞 회차 사실과 충돌
- 클리프행어 남용
- 저품질 마케팅성 문체 오염

비평 모델은 작품을 새로 쓰지 않는다. 오직 판정과 짧은 수정 지시만 반환한다.

### Gate 4. 재시도 정책

- 규칙 위반만 있으면 저비용 수정 1회
- 구조/비평 실패면 전면 재생성 1회
- 2회 이상 실패하면 그 회차는 `publish_stop`

이 상한이 없으면 완전 자동형은 비용도 터지고 품질도 흔들린다.

### Gate 5. Canon 반영 조건

다음 세 조건을 모두 만족할 때만 Canon DB 갱신을 허용한다.

- 품질 게이트 최종 통과
- 실제 플랫폼 발행 검증 성공
- 구조화 사실 추출 결과가 스키마 검증 통과

## 발행 정책 엔진

문피아와 노벨피아는 같은 작품이라도 운영 정책이 다를 수 있다. 따라서 "매일 21시 업로드" 같은 단일 스케줄러로는 부족하다.

Policy Engine은 다음을 지원해야 한다.

- 플랫폼별 발행 시간대
- 요일별 편성
- 하루 최대 발행 수
- 2연참 허용 조건
- 연속 실패 후 강제 냉각 시간
- 특정 사건 발생 시 전체 중단

예시 정책:

- 문피아: 평일 07:00 1회, 주말 2연참 허용
- 노벨피아: 매일 20:00 1회, 특별 이벤트일에만 추가 1회 허용

정책 엔진은 단순 스케줄러가 아니라 `지금 발행하는 것이 운영상 안전한가`를 판정하는 계층이어야 한다.

## 플랫폼 업로드와 검증

플랫폼 어댑터는 브라우저 자동화로 구현하되, 업로드 성공 판정은 "버튼 클릭 성공"이 아니라 `실제 생성된 회차를 다시 읽어 확인하는 것`이어야 한다.

플랫폼 자격 증명과 세션 비밀값은 프로젝트 JSON에 저장하지 않고, OS keyring 또는 동등한 보안 저장소에 둔다.

업로드 흐름:

1. 로그인
2. 작품 확인 또는 생성된 작품 매핑 불러오기
3. 회차 입력
4. 발행 옵션 설정
5. 업로드 실행
6. 회차 목록 또는 상세 페이지 재조회
7. 제목/회차 번호/예약 시간/공개 상태 확인

오류 분류:

- `retryable`
  - 일시적 DOM 실패
  - 네트워크 타임아웃
  - 저장 버튼 응답 지연
- `requires_user_action`
  - 로그인 실패
  - CAPTCHA
  - 추가 인증
  - 플랫폼 UI 구조 변경 의심
- `permanent`
  - 작품 매핑 없음
  - 정책 위반 값
  - 소스 산출물 누락

`requires_user_action`은 사건 대응형 운영과 직접 연결된다. 이 상태가 발생하면 이후 자동 발행은 일시 정지되어야 한다.

## 마케팅 패키저

마케팅은 생성 파이프라인에 붙어 있어야 하지만, 회차 본문과 절대로 같은 프롬프트에서 처리하면 안 된다. 본문 생성과 마케팅 생성을 분리해야 품질 오염과 비용 낭비를 줄일 수 있다.

Marketing Packager는 다음 산출물을 만든다.

- 플랫폼 소개문
- 회차 홍보 한 줄
- 해시태그 또는 태그 후보
- 이벤트/연참 안내 문구

제약:

- 본문 내용을 새로 추가 해석하지 않는다.
- 선정성, 과장 광고, 플랫폼 금칙어를 별도 규칙으로 검사한다.
- 발행 실패 회차에 대해서는 마케팅 산출물을 정본으로 저장하지 않는다.

## 번역 및 해외 확장 경계

향후 해외 업로드를 생각하면, 지금부터 번역 파이프라인을 한국어 본편과 분리해야 한다.

원칙:

- 한국어 정본 회차가 먼저 확정된다.
- 번역은 정본 기준으로 별도 `translation run`을 만든다.
- 용어집, 캐릭터 이름 표기, 세계관 고유명사는 번역용 사전으로 분리한다.
- 번역 품질 게이트는 원문 품질 게이트와 별개다.

`v1 권장 범위`

- 한국어 정본을 기반으로 번역 패킷 생성 인터페이스 설계
- locale별 스타일 가이드와 용어집 저장소 설계
- 해외 플랫폼 어댑터가 끼워질 공통 인터페이스 정의

`v1 비권장 범위`

- 해외 사이트까지 즉시 무인 업로드
- 국가별 마케팅 문안 자동 최적화
- 플랫폼별 상이한 법적 정책 자동 판정

이 부분은 기술적으로 가능해 보여도 운영 리스크가 커서 한국어 본편 안정화 이전에는 붙이면 안 된다.

## 비용 최적화 전략

사용자가 요구한 핵심은 가성비다. 비용은 모델 선택보다 `언제 비싼 모델을 호출하지 않을 것인가`로 관리해야 한다.

원칙:

- Planner와 규칙 검사에는 저비용 모델 또는 코드 기반 검사 우선
- Draft Writer는 중간 티어 모델 사용
- 비평 모델은 최종 후보에만 사용
- 마케팅 생성은 가장 저렴한 계층 또는 템플릿 기반 우선
- 같은 입력 스냅샷이면 캐시 재사용 가능하게 설계
- 재시도 횟수는 단계별 상한을 강제

추가 전략:

- 회차 생성에 들어가는 컨텍스트를 `Story Bible + Canon Facts + Planner Output`로 제한
- 과거 회차 전문을 다시 넣지 않음
- 발행 실패 시 무한 재생성 대신 `중단 후 인간 확인`으로 전환

## 사건 대응형 운영 모델

완전 자동형이라도 실제 운영은 `무인`이 아니라 `사건 대응형`에 가깝다. 따라서 런타임은 정상 실행보다 예외 처리에 더 많은 구조를 가져야 한다.

상태:

- `idle`
- `scheduled`
- `running`
- `cooldown`
- `paused`
- `blocked`
- `stopped`

사건 유형:

- `quality_incident`
- `platform_incident`
- `credential_incident`
- `policy_incident`
- `data_integrity_incident`

규칙:

- `quality_incident`가 임계치 이상이면 해당 작품 발행 중단
- `platform_incident`가 반복되면 플랫폼별 어댑터만 차단하고 다른 플랫폼은 유지 가능
- `credential_incident`는 즉시 사용자 개입 필요 상태로 전환
- `data_integrity_incident`는 Canon DB 갱신과 발행 둘 다 중단

이 모델이 있어야 문피아/노벨피아 중 한 곳만 장애가 나도 전체 시스템이 불필요하게 멈추지 않는다.

## 테스트와 검증 전략

이 시스템은 문장 품질만 테스트해서는 안 된다. 운영 로직 검증이 더 중요하다.

### 단위 테스트

- 정책 엔진
- 게이트 판정 로직
- 재시도 상태 전환
- Canon DB 스키마 검증
- 플랫폼 결과 집계기

### 통합 테스트

- 회차 1건의 end-to-end 런 시뮬레이션
- Gate 실패 후 중단 시나리오
- 플랫폼별 부분 성공 시나리오
- 발행 성공 후 Canon 반영 시나리오

### 회귀 테스트

- 인물명 오타 사전
- 회차 번호 어긋남
- 반복 개그 인플레 검출
- 마케팅 금칙어 검출

### 운영 검증

- 실제 플랫폼 테스트 계정으로 smoke test
- 저장된 `run snapshot` 재생성 가능성 검증
- 장애 발생 시 `stopped` 또는 `paused` 전환 확인

## 현실적인 한계와 제약

이 설계는 완전 자동형을 목표로 하지만, 다음 한계를 정면으로 인정해야 한다.

- 문학적 품질은 보장할 수 없고, 오직 기준 미달을 줄일 수 있을 뿐이다.
- 브라우저 자동화는 플랫폼 UI 변경에 취약하다.
- 하루 2연참은 기술 문제가 아니라 품질 저하와 비용 증가를 동시에 유발한다.
- 번역 자동화는 한국어 본편보다 품질 편차가 더 크다.
- 장기 연재는 결국 드리프트와 인플레를 유발하므로, 구조화 사실과 중단 정책이 없으면 반드시 무너진다.

따라서 이 시스템의 성공은 "얼마나 자주 잘 썼는가"보다 "어떤 조건에서 자동으로 멈추는가"로 평가해야 한다.

## 권장 구현 순서

이 설계는 한 번에 전부 구현하기보다 다음 순서로 가야 한다.

1. Story Bible / Canon DB / Run Snapshot 구조 확립
2. Episode Planner / Draft Writer / Quality Gate Orchestrator 구축
3. Publish Packager / Munpia / Novelpia Adapter 연동
4. Release Policy Engine과 사건 대응형 상태 기계 도입
5. Marketing Packager 추가
6. Translation Boundary와 번역 패킷 저장소 추가

이 순서를 뒤집으면, 발행 자동화가 생성 품질 문제를 숨겨버리거나, 번역 확장이 본편 안정화를 방해하게 된다.

## 결론

권장안은 `다단계 품질 게이트형 완전 자동 연재 시스템`이다. 핵심은 회차 생성 자체가 아니라, `구조화 사실 기반 컨텍스트`, `발행 전 품질 게이트`, `플랫폼별 정책 엔진`, `사건 대응형 운영 상태`, `발행 성공 회차만 정본으로 반영하는 저장 구조`에 있다.

즉, 새 시스템은 "웹소설 자동 생성기"가 아니라 다음과 같이 정의하는 것이 맞다.

`저비용으로 회차를 생산하고, 강하게 걸러내고, 안전할 때만 발행하며, 실패하면 즉시 멈추는 자동 연재 운영 시스템`
