# 외부 플랫폼 업로드 설계

**목표**

현재 앱에 `[6] 외부 플랫폼 업로드` 탭을 추가해 문피아와 노벨피아에 대한 자동 로그인, 신규 작품 생성, 회차 업로드, 발행 옵션 설정, 예약 업로드를 지원한다. 이 기능은 기존 `[5] 자동화 연재 모드`와 분리된 큐, 스케줄러, 실행 상태, 이력을 가져야 하며, 로컬에 저장된 회차 파일을 외부 플랫폼에 배포하는 책임만 가진다.

**배경**

현재 앱은 [ui/app.py](c:/Users/W/novel_autowriter_1/ui/app.py), [ui/automation.py](c:/Users/W/novel_autowriter_1/ui/automation.py), [core/automation_runtime.py](c:/Users/W/novel_autowriter_1/core/automation_runtime.py), [core/automation_store.py](c:/Users/W/novel_autowriter_1/core/automation_store.py)를 중심으로 프로젝트별 자동화 큐와 스케줄러를 이미 갖고 있다. 이 구조는 `원고 생성`과 `외부 업로드`를 같은 런타임에 섞기보다, 저장 구조와 스케줄링 패턴은 재사용하고 책임은 분리하는 데 적합하다.

사용자는 문피아와 노벨피아에 연재를 진행할 예정이며, 1차 버전에서 다음 범위를 원한다.

- ID/비밀번호 기반 자동 로그인
- 신규 작품 생성
- 기존 또는 신규 작품에 대한 회차 업로드
- 발행 옵션 설정
- 예약 시간에 따라 자동 업로드되는 업로드 큐
- 큐 항목마다 문피아, 노벨피아, 또는 양쪽 동시 선택

또한 `[5]`와 `[6]`은 반드시 분리되어야 한다. `[5]`는 글 생성과 로컬 저장만 담당하고, `[6]`은 로컬 파일을 읽어 외부 플랫폼에 업로드만 담당한다.

## 접근 방식

### 옵션 1: `[5]`와 `[6]`을 분리하고 업로드만 독립 실행

`[5]`는 기존 그대로 `원고 생성 큐`를 유지하고, `[6]`은 별도 `외부 업로드 큐`를 가진다. `[6]`은 프로젝트 폴더에 저장된 회차 파일을 읽어 문피아/노벨피아 업로드를 수행한다.

장점:
- 실패 원인과 복구 경로가 명확하다.
- 문피아/노벨피아 로그인 만료나 사이트 변경이 `[5]`의 생성 흐름을 오염시키지 않는다.
- 기존 자동화 구조를 그대로 재사용하면서 책임 경계를 유지할 수 있다.

단점:
- 사용자 입장에서 자동화 기능이 탭 둘로 나뉜다.
- 동일 회차를 생성하고 업로드 큐에 넣는 동작이 논리적으로 이어지지만 실행 런타임은 따로 관리해야 한다.

### 옵션 2: `[5]`에 외부 업로드까지 통합

원고 생성 후 바로 외부 업로드까지 하나의 큐에서 처리한다.

장점:
- 사용자 시점에서는 자동화 입구가 하나다.

단점:
- `생성 실패`, `검수 실패`, `로그인 실패`, `업로드 실패`, `플랫폼별 부분 성공`이 한 런타임에 뒤섞인다.
- 재시도 기준이 과도하게 복잡해진다.
- 기존 자동화 UI와 저장 구조가 빠르게 비대해진다.

### 옵션 3: 별도 외부 업로드 앱 또는 스크립트로 분리

현재 앱 외부에 별도 배포 자동화 도구를 만든다.

장점:
- 배포 도메인만 독립적으로 확장 가능하다.

단점:
- 프로젝트 설정, 회차 경로, 계정 관리, 로그가 이중화된다.
- 현재 앱과의 연결성이 약해진다.

## 추천안

옵션 1을 채택한다. 즉, `[5]`와 `[6]`은 완전히 분리하고, `[6]`은 프로젝트 폴더에 저장된 회차 파일만 참조하는 독립 업로드 시스템으로 설계한다.

`연결`의 의미는 런타임 통합이 아니라 산출물 참조다. 예를 들어 `[5]`가 `12화.md`를 저장하고, `[6]`은 그 파일을 선택해 `2026-03-14 21:00`에 문피아와 노벨피아로 업로드한다. 업로드 실패는 `[6]`에서만 관리되며 `[5]`의 생성 이력과 상태에는 영향을 주지 않는다.

## 저장 구조와 데이터 모델

프로젝트별로 별도 업로드 저장소를 둔다.

- `data/projects/<project>/publishing/config.json`
- `data/projects/<project>/publishing/queue.json`
- `data/projects/<project>/publishing/runtime.json`
- `data/projects/<project>/publishing/history.jsonl`

역할은 다음과 같다.

- `config.json`: 업로드 활성화 여부, 스케줄, 재시도 정책, 브라우저 모드, 플랫폼별 기본 설정, 작품 매핑 정보
- `queue.json`: 예약된 업로드 작업 목록
- `runtime.json`: 현재 상태, 현재 작업 ID, 마지막 실행 시각, 마지막 오류
- `history.jsonl`: 업로드 시도 결과 기록

민감정보는 JSON에 저장하지 않는다. 문피아와 노벨피아의 ID/비밀번호는 `keyring`에 저장하고, 설정 파일에는 활성화 여부와 플랫폼별 기본 발행 옵션, 작품 ID만 저장한다.

예시 `config.json`:

```json
{
  "enabled": true,
  "schedule": {
    "type": "daily",
    "time": "21:00",
    "days": [],
    "hours": 24
  },
  "retry_policy": {
    "max_attempts": 2
  },
  "browser": {
    "headless": false
  },
  "platforms": {
    "munpia": {
      "enabled": true,
      "work_id": "",
      "default_publish_visibility": "public",
      "default_age_grade": "general"
    },
    "novelpia": {
      "enabled": true,
      "work_id": "",
      "default_publish_visibility": "public",
      "default_age_grade": "general"
    }
  }
}
```

업로드 큐 항목은 하나의 회차와 다수의 대상 플랫폼을 표현한다. 큐 항목 하나가 문피아와 노벨피아를 동시에 가질 수 있으므로, 상위 상태와 플랫폼별 하위 상태를 모두 가진다.

예시 `queue.json` 항목:

## 작품 신규 생성 모델

작품 신규 생성은 회차 업로드와 같은 큐에 넣지 않고, 플랫폼별 작품 매핑을 만드는 별도 작업으로 취급한다.

흐름:

1. 사용자가 `[6]` 탭에서 플랫폼 계정과 작품 메타데이터를 입력한다.
2. 문피아 또는 노벨피아 작품 생성 버튼을 누른다.
3. 플랫폼별 자동화가 신규 작품 생성 화면을 열고 폼을 채운다.
4. 생성 성공 시 작품 식별자(`work_id`)를 추출해 `config.json`에 저장한다.
5. 이후 업로드 큐는 해당 `work_id`를 사용해 회차를 발행한다.

이렇게 하면 작품 생성은 플랫폼별로 한 번만 수행되고, 한 플랫폼만 먼저 성공한 경우도 정상적으로 유지된다.

## 실행 흐름과 스케줄 처리

`[6]`은 `[5]`와 동일하게 백그라운드 폴링 방식으로 동작하되, 업로드 큐만 감시한다.

런타임 흐름:

1. `publishing/config.json`에서 활성화 여부와 스케줄을 읽는다.
2. 현재 시각이 업로드 가능한 시점인지 판정한다.
3. `queue.json`에서 `pending` 또는 `partial_failed` 작업 중 가장 먼저 처리할 항목을 선택한다.
4. 소스 파일 존재 여부를 확인하고 본문을 로드한다.
5. 대상 플랫폼마다 순차적으로 로그인, 작품 확인, 회차 업로드, 발행 옵션 적용을 수행한다.
6. 플랫폼별 결과를 저장하고 전체 상태를 계산한다.
7. `runtime.json`과 `history.jsonl`을 갱신한다.

플랫폼 순서는 고정하고, 한 플랫폼이 실패해도 다른 플랫폼은 계속 시도한다. 둘 다 실패하면 `failed`, 하나만 실패하면 `partial_failed`, 둘 다 성공하면 `done`이다.

## 발행 옵션 처리

발행 옵션은 큐 항목의 플랫폼별 타겟 설정에 둔다. 지원 범위는 다음과 같다.

- 즉시 발행
- 예약 발행
- 공개/비공개
- 플랫폼별 기본값 override

`queue.scheduled_at`은 앱 내부 업로드 작업 시작 시각이고, `targets.<platform>.reserved_at`은 사이트 내부 예약 발행 시각이다. 두 개념은 반드시 분리한다. 예를 들어 앱은 `2026-03-14 20:55`에 사이트에 접속해 업로드를 수행하고, 사이트 내부 발행 시각은 `2026-03-14 21:00`으로 넣을 수 있어야 한다.

## 로그인과 세션 관리

1차 버전은 쿠키 저장 재사용보다 단순한 로그인 방식으로 간다.

- 작업 실행 시마다 ID/비밀번호로 로그인 시도
- 같은 작업 내에서는 세션 유지
- 작업 종료 시 브라우저 종료
- 다음 작업에서 다시 로그인

이 방식은 실행 시간이 조금 더 길지만, 세션 저장 보안 문제를 피하고 디버깅이 쉽다. CAPTCHA나 추가 인증이 뜨면 해당 작업은 사용자 개입이 필요한 실패로 기록한다.

## 브라우저 자동화 기술

브라우저 자동화는 Playwright를 사용한다.

이유:

- 로그인 모달, 동적 DOM, SPA 기반 화면 대응이 쉽다.
- 수동 디버깅과 재현이 편하다.
- 향후 사이트별 클라이언트 구현을 공통 인터페이스로 감싸기 쉽다.

1차 구현은 저장된 브라우저 상태 재사용을 하지 않고, `브라우저 시작 -> 로그인 -> 작업 수행 -> 종료` 패턴으로 고정한다.

## 에러 분류와 재시도 정책

실패는 세 종류로 나눈다.

- `retryable`
  - DOM 로딩 지연
  - 네트워크 타임아웃
  - 일시적 요소 탐색 실패
- `requires_user_action`
  - 로그인 실패
  - CAPTCHA
  - 추가 인증
  - 필수 메타데이터 누락
- `permanent`
  - 소스 파일 없음
  - 플랫폼 미선택
  - 지원하지 않는 발행 옵션

정책:

- `retryable`만 자동 재시도
- `requires_user_action`은 런타임을 `paused`로 전환
- `permanent`는 즉시 `failed`

## UI 설계

`[6] 외부 플랫폼 업로드` 탭은 다음 네 블록으로 구성한다.

### 1. 플랫폼 계정/작품 설정

- 문피아 ID/PW 저장
- 노벨피아 ID/PW 저장
- 플랫폼 활성화 토글
- 작품 신규 생성 버튼
- 생성된 작품 ID 표시
- 플랫폼별 기본 발행 옵션 설정

### 2. 업로드 스케줄 설정

- daily / weekly / interval
- 브라우저 모드(headless / headed)
- 재시도 횟수
- 자동 업로드 활성화

### 3. 업로드 큐

- 로컬 회차 파일 선택
- 회차 제목 입력 또는 수정
- 대상 플랫폼 체크박스
- 플랫폼별 발행 옵션
- 예약 시각 설정
- 큐 추가 / 순서 이동 / 재시도 / 삭제

### 4. 실행 상태와 이력

- 현재 상태
- 현재 작업
- 마지막 오류
- 최근 업로드 이력
- 수동 1회 실행 버튼
- paused 해제 버튼

이 구조는 [ui/automation.py](c:/Users/W/novel_autowriter_1/ui/automation.py)의 사용감을 최대한 유지하면서 외부 배포 설정만 별도로 노출한다.

## 구현 단위

권장 파일 구조는 다음과 같다.

- `ui/publishing.py`
  - `[6]` 탭 렌더링
  - 설정/큐/상태 화면 처리
- `core/publishing_store.py`
  - publishing 저장소 로드/저장
- `core/publishing_runtime.py`
  - 업로드 스케줄 실행, 상태 계산, 이력 기록
- `core/platform_clients/base.py`
  - 플랫폼 공통 인터페이스
- `core/platform_clients/munpia.py`
  - 문피아 로그인, 작품 생성, 회차 업로드
- `core/platform_clients/novelpia.py`
  - 노벨피아 로그인, 작품 생성, 회차 업로드
- `core/platform_credentials.py`
  - keyring 자격증명 저장/조회
- `core/chapter_source.py`
  - 회차 파일 읽기와 본문 정리

사이트 변경은 클라이언트 레이어에 가두고, UI와 런타임은 플랫폼 공통 계약만 사용하도록 한다.

## 테스트와 검증 방향

실사이트 E2E는 기본 자동 테스트에 넣지 않는다. 대신 아래 조합으로 간다.

1. 단위 테스트
- 저장소 기본값, 로드, 저장
- 큐 상태 계산
- 재시도/paused 판정
- 플랫폼 결과 집계

2. UI 헬퍼 테스트
- `[6]` 탭 라벨 추가
- 상태 요약 문자열
- 큐 row 변환
- 설정 요약 포맷

3. 수동 검증 체크리스트
- 문피아 로그인 성공
- 노벨피아 로그인 성공
- 문피아 작품 신규 생성
- 노벨피아 작품 신규 생성
- 기존 작품 회차 업로드
- 부분 성공 시 `partial_failed`
- 예약 발행 옵션 반영
- 로그인 실패 시 `paused`

실사이트 자동 테스트는 계정, CAPTCHA, 정책 팝업, 사이트 상태에 영향을 크게 받으므로 CI 기본 테스트 대상으로 두지 않는다.

## 범위 제외

1차 버전에서는 다음을 하지 않는다.

- SNS 로그인 세션 재사용
- 쿠키/스토리지 상태 저장
- 표지 업로드를 위한 이미지 편집 도구
- 플랫폼별 고급 메타데이터 최적화
- 업로드 직후 통계 수집
- 플랫폼별 동시 병렬 업로드

핵심 목표는 `[5]`와 분리된 안정적인 업로드 큐와, 문피아/노벨피아 신규 작품 생성 및 회차 발행 자동화를 구현하는 것이다.
